# Temporal для сценариев: зачем это инвестору

## Проблема

Оркестрация многошаговых сценариев (агент + инструменты + внешние API) в одном процессе ломается при рестартах, пиках нагрузки и длительных паузах: состояние «живёт» только пока жив сервер.

## Решение

**Temporal** даёт отдельный слой надёжного выполнения workflow: история шагов, автоматические повторы activity, возобновление после сбоев воркера и прозрачная наблюдаемость без ручного восстановления цепочки.

## Что получает продукт

- **Долговечность**: сценарии переживают деплой API и падение процесса.
- **Предсказуемость**: политики retry и таймауты на уровне платформы, а не разрозненного кода.
- **Масштабирование**: воркеры можно наращивать горизонтально; очередь и история — у Temporal.
- **Соответствие рискам**: идемпотентность tool-activity и дедуп результатов снижают двойные списания и побочные эффекты при повторах.

## Когда достаточно in-memory

Для быстрых демо, локальной разработки и тестов без инфраструктуры — встроенный in-memory оркестратор остаётся простым путём; Temporal включается осознанно для сред, где важна устойчивость и SLA.

## Сводка

Temporal — это **страховка и масштаб** для сценариев с деньгами, данными и длительными цепочками: меньше ручного восстановления, больше контроля при росте нагрузки.
